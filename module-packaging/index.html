What is a Python package?

Deconstructing that definition:

  • When I write import modulename I am loading a module
  • The module can contain "arbitrary Python objects"
    • That could iclude other modules!
    • It's almost like we need a special name for "a module that can contain other modules"!

What is a Python package?

Deconstructing that definition:

  • When I write import modulename I am loading a module
  • The module can contain "arbitrary Python objects"
  • When I load the module, its name gets added to the namespace
    • namespace: a collection of currently defined names and they objects they reference
    • So now I can use the module name to access things inside of it with dot notation, like another module that contains a function: numpy.random.default_rng

Aside: why does Python have modules?

Because it's good for your code to be modular.

Trick question: the word "package" is overloaded in Python

We'll fix this by talking about import packages and distribution packages See "Distribution package vs import package" on the Python Packaging User Guide.

Import package

  • A Python module that you can import in your code. Usually this is what we mean hen we say "Python package".
  • An import package is a module that can contain other modules. Think of it as a toolbox full of tools you use in your own code.

To make it so that you and other people can import your own code, you make a directory with a specific structure, that we're going to teach you today.

Distribution package

  • A piece of software you can install. Other languages and even operating systems also have distribution packages, that you install with a package manager. This is the kind of package you get when you write pip install pkg.
  • To make it so that other people can pip install your import package, you have to build a distribution package.

When should I make a package?

When might I make an (import) package?

  • Code for a paper
  • A set of common functions you use across projects
  • A generalized tool that other researchers can use

When should I make an (distribution) package?

  • Code for a paper
  • A generalized tool that other researchers can use

rescale function from Software Carpentry:


                            import numpy as np

                            def rescale(input_array):
                                """Rescales an array from 0 to 1.

                                Takes an array as input, and returns a corresponding array scaled 
                                so that 0 corresponds to the minimum and 1 to the maximum value 
                                of the input array.
                                """
                                L = np.min(input_array)
                                H = np.max(input_array)
                                output_array = (input_array - L) / (H - L)
                                return output_array

Call it:


                            rescale(np.linspace(0, 100, 5))
                        

                            array([ 0.  ,  0.25,  0.5 ,  0.75,  1.  ])

compphys contents:


  • __init__.py: tells Python this is a package; typically empty. Executed before any other modules imported.
  • Two modules: constants.py and physics.py
  • more is a submodule
  • assets is a subdirectory; not a submodule since it doesn't have __init__.py (orphan.py is unreachable!)
  • tests: subdirectory with tests

                            touch README.md

A README is a form of software documentation, and should contain at minimum:

  • the name of your software package
  • a brief description of what your software does or provides
  • installation instructions
  • a brief usage example
  • the type of software license (with more information in a separate LICENSE file, described next)

In addition, a README may also contain:

  • badges near the top that quickly show key information, such as the latest version, whether the tests are currently passing
  • information about how people can contribute to your package
  • a code of conduct for people interacting around your project (in GitHub Issues or Pull Requests, for example)
  • contact information for authors and/or maintainers

Keep a CHANGELOG


Over time, your package will change as you fix bugs, add features, and make improvements.

Technically, your Git history should contain all the records of changes, but in practice this isn't very helpful for functionality changes.

Instead, keep a CHANGELOG that is a human-readable record of major changes between each version.
